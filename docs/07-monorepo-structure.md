# 07 — Final monorepo structure

pnpm workspaces + Turborepo. The CMS / builder / contracts / adapters separation is preserved.
Packages are created **when they are implemented** (no empty folders); this is the target
structure and the order of appearance.

```
astrocms/
├── apps/
│   ├── cms-server/          # Fastify: API v1, serves static admin/builder, webhooks
│   ├── cms-admin/           # React SPA: TanStack Router + Query + RHF + Zod
│   ├── builder-app/         # React SPA that mounts builder-react (route /admin/builder)
│   ├── astro-demo/          # sample Astro site + preview route + block registry
│   └── docs/                # browsable documentation (Astro Starlight)  [optional for MVP]
│
├── packages/
│   ├── contracts/           # types + Zod: API, document, manifest, iframe messages
│   ├── schemas/             # field system (descriptor → Zod/TS/form/defaults)
│   ├── storage/             # StorageDriver + fs/s3/r2/minio drivers
│   ├── ui/                  # panel's React design system (accessible)
│   ├── testing/             # fixtures, factories, test DB helpers
│   │
│   ├── cms-core/            # CMS use cases (no HTTP)
│   ├── cms-database/        # Drizzle schema + migrations + repos
│   ├── cms-auth/            # sessions, hashing, CSRF, RBAC
│   ├── cms-sdk/             # TS client for API v1
│   ├── cms-astro/           # Astro integration (SSR helpers, preview, manifest loader)
│   │
│   ├── builder-core/        # document, commands, undo/redo, validation, migrations
│   ├── builder-react/       # canvas, tree, inspector, dnd (on top of builder-core)
│   ├── builder-astro/       # block render runtime + defineBuilderConfig + manifest
│   ├── builder-sdk/         # createBuilderClient + BuilderStorageAdapter
│   ├── builder-default-blocks/  # 10 base blocks (def + reference Astro components)
│   ├── builder-adapters/    # cmsBuilderAdapter, inMemoryAdapter, jsonFileAdapter
│   │
│   ├── cli/                 # `astrocms` CLI: generate/add/migrate/validate/manifest (JSON output)
│   └── mcp/                 # MCP server: exposes the CMS/project as AI tools
│
├── docs/                    # THIS set of design documents (Phase 0)
├── migrations/              # SQL generated by drizzle-kit (reproducible source)
├── docker/                  # Dockerfiles (cms-server, astro-demo)
├── docker-compose.yml       # postgres, minio, cms-server, astro-demo
├── turbo.json
├── pnpm-workspace.yaml
├── tsconfig.base.json
├── package.json
├── .npmrc
├── .gitignore
└── .env.example
```

## Changes from the original proposal (justified)

- **`builder-adapters` as its own package** (instead of inside `builder-sdk`): the
  `cmsBuilderAdapter` adapters depend on `cms-sdk`; keeping them separate prevents `builder-sdk`
  (the agnostic core) from carrying a CMS dependency. This reinforces portability.
- **`cms-astro` unifies** "Astro SDK + SSR integration + preview": it is Astro's face toward the CMS.
- **`apps/docs` optional for the MVP**: it does not block the success criterion; it can be added in Phase 7.
- **`registry.generated.ts`** (`type → component` map) is generated during `astro-demo`'s build; it is
  not a package, it's a local artifact of the Astro project.

## Package names (npm scope)

`@astrocms/contracts`, `@astrocms/schemas`, `@astrocms/storage`, `@astrocms/ui`,
`@astrocms/testing`, `@astrocms/cms-core`, `@astrocms/cms-database`, `@astrocms/cms-auth`,
`@astrocms/cms-sdk`, `@astrocms/cms-astro`, `@astrocms/builder-core`, `@astrocms/builder-react`,
`@astrocms/builder-astro`, `@astrocms/builder-sdk`, `@astrocms/builder-default-blocks`,
`@astrocms/builder-adapters`, `@astrocms/cli`, `@astrocms/mcp`.

Infrastructure adapters (storage/cache/queue/mailer/ai) live alongside their port:
`storage` already includes its own; `cache`/`jobs`/`mail`/`ai` are subpackages of `cms-core` with
the port + default adapter (0 infra), and optional adapters (redis, bullmq, smtp…) are
added when a deployment requires them (see [ADR-0008](adr/0008-infrastructure-agnostic.md)).

## Conventions

- **Strict TS** in `tsconfig.base.json` (`strict`, `noUncheckedIndexedAccess`,
  `exactOptionalPropertyTypes`). No `any` except where documented and justified at the point of use.
- **ESLint + Prettier** at the root; a **boundaries** rule (`dependency-cruiser`) that forbids
  `builder-*` from importing from `cms-core`/`cms-database`/`cms-auth`.
- **Turbo pipelines:** `build`, `dev`, `lint`, `typecheck`, `test`, `test:e2e`, `db:migrate`, `db:seed`.
- **Each package** exposes typed `exports` and its own `tsconfig` that extends the base one.
- **No secrets in the repo**: `.env.example` documents variables; actual values go in a gitignored `.env`.
